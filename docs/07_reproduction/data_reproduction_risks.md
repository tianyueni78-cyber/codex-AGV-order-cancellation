# 数据层复现风险

## 核心问题

复现实验首先要保证：**同一个算例、同一组机器/AGV参数、同一个 seed，能够生成可追踪的同一类实验结果。**

当前最大风险不是算法缺失，而是数据入口、路径、副作用和参数记录尚未完全工程化。

## 关键风险

| 风险 | 表现 | 对复现的影响 | 整理方向 |
|---|---|---|---|
| 自动生成中间文件 | `benchmarkRead.m` 会保存 `data.mat` | 不清楚当前结果使用的是哪次生成的数据 | 用 `read_fjsp.m` 返回结构体，避免隐式写文件 |
| 输入文件被修改 | `distance_from_xy.m` 会写回机器 Excel | 原始输入不再稳定 | 计算结果写入 `outputs/`，输入文件只读 |
| 路径依赖 | 多处使用当前目录和 `cd(...)` | 换启动目录后结果不可复现 | 统一以项目根目录构造相对路径 |
| 参数分散 | 主脚本和算法内部都设置参数 | 论文参数表难以对应真实运行 | 建立 experiment config |
| 随机过程未固定 | `rand`、`randperm`、`randn` 多处出现 | 每次运行结果不同且不可追踪 | 每次独立运行记录 `rng(seed)` |
| 输出追加/覆盖 | `results.txt` append，图像按固定名字保存 | 多次实验结果混杂或覆盖 | 每次实验建立独立输出目录 |

## read_fjsp 纯函数化的意义

新增 `src/data/read_fjsp.m` 的价值不在于改变算法，而在于建立一个稳定的数据入口：

```text
.fjs 文件 -> problem 结构体 -> 算法输入
```

它避免了旧流程中的隐式副作用：

```text
.fjs 文件 -> benchmarkRead.m -> data.mat -> 主脚本 load
```

对复现来说，这一步意味着：

- 输入和输出边界更清楚。
- `.fjs` 读取可以被单独测试。
- 不会因为运行读取函数而覆盖 `data.mat`。
- 后续可以把 `problem` 作为统一实验对象传递给算法。

## 必须配置化的数据与参数

优先配置化：

- 算例路径：`.fjs` 文件位置。
- 机器数据路径：机器 Excel 文件与 sheet 名。
- AGV 数据路径：AGV Excel 文件与 sheet 名。
- 距离矩阵来源：直接读取或由坐标计算。
- 能耗参数：机器加工/空载能耗，AGV 空载/负载能耗。
- AGV 参数：数量、速度挡位、电池容量、充电阈值、充电速度。
- 算法参数：种群规模、迭代次数、交叉变异概率、VNS/Q-learning 参数。
- 评价参数：HV 参考点、归一化方式、独立运行次数。

## 最小复现实验

最小复现实验不应从完整算法对比开始，而应分层验证：

1. 读取 `.fjs`，得到非空 `problem`。
2. 读取机器和 AGV 小样本数据。
3. 固定 `rng(seed)`。
4. 构造一个小种群。
5. 对一个染色体运行 `fitness`。
6. 确认 `makespan` 和 `total energy` 非空且可追踪。
7. 再运行少量代数算法。
8. 所有结果写入 `outputs/实验名/`。

当前 `tests/test_read_fjsp.m` 覆盖第 1 步，并验证不会生成项目根目录 `data.mat`。

## 工程化优先级

数据层应先于算法层整理。原因是：

- 论文可信度取决于实验输入是否清楚。
- 算法对比依赖同一套数据输入。
- `sorting` 和 `fitness` 对数据结构强依赖，数据不稳定会直接改变目标值。
- 配置化之前，结果很难与论文参数表一一对应。

下一步工程化应优先处理：机器/AGV Excel 读取纯函数化、实验配置文件、输出目录规范、随机 seed 记录。
