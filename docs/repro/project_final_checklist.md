# 项目最终收口 Checklist

> 用途：给导师做最后验收时，快速判断这个项目是否已经达到“范围冻结、证据齐全、结论封版、后续不再改主线”。

## 1. 冻结边界是否清楚

- [ ] README 已明确写成“已冻结收口版本”，而不是继续推进中的阶段稿。
- [ ] 总收口文档明确说明主线已经冻结。
- [ ] 冻结边界与非范围已经单独列出。
- [ ] 机器故障、AGV 故障、新订单插入、多订单连续取消、强化学习都没有被当成当前主线继续推进。

## 2. 证据入口是否收口

- [ ] 有一份总证据索引，能把导师带到最该看的文件。
- [ ] 文件入口已经覆盖：总收口文档、冻结边界索引、baseline、traceability、大批量结果、交付清单。
- [ ] 导师不需要翻很多阶段文档，也能直接定位关键证据。
- [ ] 最终可引用的输出 CSV 已明确列出。

## 3. baseline 是否封口

- [ ] baseline 已明确说明它是“对照线”，不是全部输出。
- [ ] baseline 已说明对比对象是 `raw_code/`、`auto_selection`、`local_only`、`complete_only`。
- [ ] baseline 已说明主结论和辅助验证分别是什么。
- [ ] baseline 已明确不能推出全局最优、全量泛化和论文终结论。

## 4. 主线叙事是否收住

- [ ] README 的主要叙事已经从“阶段推进”改成“已冻结、可复现、可引用”。
- [ ] 阶段文档已经降级为历史材料或扩展预案。
- [ ] 不再有把正式搜索、多随机种子长实验、30 秒搜索入口塞进一键收口的暗示。
- [ ] 后续只保留文档修订、证据补齐或版本封版，不再改主算法链路。

## 5. 对外汇报是否够用

- [ ] 一句话能讲清楚项目做什么。
- [ ] 一句话能讲清楚项目不做什么。
- [ ] 一句话能讲清楚现在为什么可以收口。
- [ ] 一句话能讲清楚哪些结果最值得看。

## 6. 最终判定

满足下面 4 条，就可以按“完整代码包、已冻结可交付版”来汇报：

1. 范围冻结。
2. 证据齐全。
3. 结论封版。
4. 后续不再改主线。

如果这 4 条都成立，这个项目就不再是研发中的阶段稿，而是一个可以直接展示和复现的交付代码包。
