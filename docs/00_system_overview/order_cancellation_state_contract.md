# 订单取消状态分类契约

> 本文档已不再作为阶段 B 的主要阅读入口。阶段 B 的事件结构、状态分类、测试入口和 smoke 结果已整合到 [阶段 B 工作记录](stage_b_work_record.md)。本文档仅保留为历史过程记录。

本文档定义阶段 B 的订单取消状态分类规则。后续状态提取函数必须先遵守本契约，再进入局部修复或完全重调度。

## 1. 适用范围

本契约只处理单个已知取消事件：

```matlab
cancel.job_id
cancel.cancel_time
cancel.policy
```

第一版支持策略：

```text
cancel_unstarted_operations_only
```

阶段 B 只做状态识别和取消任务列表提取，不生成新调度，不修改 `machineTable` 或 `AGVTable`。

## 2. 工序时间字段约定

每道工序必须能从正常调度结果中得到：

1. 工件编号 `job_id`。
2. 工序编号 `operation_id`。
3. 开始时间 `start`。
4. 结束时间 `end`。
5. 所属机器编号。

若后续实现从 `machineTable` 读取字段，应先在代码中统一转换为上述逻辑字段，再按本文档分类。

## 3. 状态分类规则

设 `t = cancel.cancel_time`。

| 状态 | 判定条件 | 含义 |
|---|---|---|
| 已完成 | `end <= t` | 取消发生时，该工序已经完成加工 |
| 正在加工 | `start < t && t < end` | 取消发生时，该工序正在机器上加工 |
| 尚未开工 | `start >= t` | 取消发生时，该工序还没有开始加工 |

以上三类状态互斥且覆盖所有合法工序时间区间。

## 4. 边界条件

边界条件按离散事件语义处理：

1. `t == end`：工序视为已完成。
2. `t == start`：工序视为尚未开工。
3. `start < t < end`：工序视为正在加工。

因此，只有取消时刻严格落在加工区间内部时，才判定为正在加工。

## 5. 被取消订单的处理规则

对于 `job_id == cancel.job_id` 的工序：

| 状态 | 第一版处理 |
|---|---|
| 已完成 | 保留为历史执行记录，不删除 |
| 正在加工 | 标记为不支持，阶段 B 只报告该状态 |
| 尚未开工 | 列入取消任务列表 |

第一版策略 `cancel_unstarted_operations_only` 的含义是：只取消尚未开工的工序；已经完成的工序作为历史保留；正在加工的工序不在第一版中强行中断。

## 6. 第一版不支持的情况

以下情况在阶段 B 第一版中必须明确报告为不支持或非法输入，不应继续进入候选重调度：

1. 被取消订单存在正在加工工序。
2. 同一事件中包含多个被取消订单。
3. `cancel_time` 未知、随机或为时间区间。
4. 取消事件同时伴随机器故障或 AGV 故障。
5. 取消后立即插入新订单。
6. 工序缺少可识别的开始时间或结束时间。
7. 工序时间区间非法，例如 `end < start`。

## 7. 后续阶段复用方式

阶段 B 的状态提取结果应至少支持后续读取：

1. 已完成工序列表。
2. 正在加工工序列表。
3. 尚未开工工序列表。
4. 被取消订单的尚未开工工序列表。
5. 是否存在第一版不支持的正在加工取消工序。

阶段 C 只允许删除“被取消订单的尚未开工工序”。阶段 D 也必须以该列表作为排除对象，不能删除已完成历史工序。

## 8. 验收标准

1. 状态分类规则清楚区分已完成、正在加工和尚未开工。
2. `cancel_time == start` 和 `cancel_time == end` 的边界语义明确。
3. 被取消订单的已完成工序、正在加工工序、尚未开工工序处理方式明确。
4. 第一版不支持的情况已经列出，后续代码不得静默忽略。
