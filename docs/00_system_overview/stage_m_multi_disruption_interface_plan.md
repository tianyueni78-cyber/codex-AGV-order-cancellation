# 阶段 M：多扰动扩展接口

本文档是阶段 M 的主文档。

## 1. 阶段目标

阶段 M 只定义多扰动扩展接口的边界，明确订单取消、机器故障、AGV 故障三类事件的职责范围。

当前主线仍以订单取消为核心，只处理任务集合变化；机器故障和 AGV 故障仅保留未来扩展边界。

## 2. 事件边界

本阶段只定义三类事件的边界，不把它们混进同一个实现入口。

- 订单取消：任务集合变化。
- 机器故障：加工资源可用性变化。
- AGV 故障：运输资源可用性变化。

当前订单取消主线只处理任务集合变化，不实现机器故障或 AGV 故障。

后续若扩展机器故障或 AGV 故障，应单独开启新阶段，并保持与当前订单取消实现隔离。

## 3. 状态提取边界

### 3.1 订单取消

- 提取被取消订单。
- 提取受影响工序/运输任务。
- 区分已完成、执行中、未开始任务。
- 订单取消属于任务集合变化。

### 3.2 机器故障

- 未来提取故障机器。
- 未来提取故障时刻。
- 未来提取维修区间。
- 未来提取受影响加工任务。
- 机器故障属于加工资源可用性变化。

### 3.3 AGV 故障

- 未来提取故障 AGV。
- 未来提取故障时刻。
- 未来提取维修区间。
- 未来提取受影响运输任务。
- AGV 故障属于运输资源可用性变化。

## 4. 修复与重调度入口边界

三类事件应进入各自独立的修复与重调度入口。

- 订单取消入口：只处理任务集合变化。
- 机器故障入口：未来只处理加工资源可用性变化。
- AGV 故障入口：未来只处理运输资源可用性变化。

当前阶段只保留边界说明，不实现机器故障或 AGV 故障入口。

当前订单取消主线不得混入机器故障或 AGV 故障的状态、修复、重调度逻辑。

## 5. 停止条件

阶段 M 到边界说明完成即停止。

- 不修改算法代码。
- 不实现机器故障。
- 不实现 AGV 故障。
- 不把三类事件混入同一个实现入口。
- 后续机器故障/AGV 故障扩展应单独开新阶段，不污染当前订单取消实现。
- 后续若继续扩展，先确认事件类型和入口边界。
