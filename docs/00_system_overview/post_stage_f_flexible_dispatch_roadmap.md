# 阶段 F 之后路线图：订单取消灵活调度

本文档使用 `define-goal` 的方式，定义阶段 F 之后如何继续推进，使项目从“单订单取消第一版闭环”逐步扩展到“订单取消问题可以灵活调度”的程度。

A-F 不是整个项目的全部阶段。A-F 的定位是第一版最小闭环：

```text
单订单取消
已知取消时刻
正常计划作为基线
局部修复候选
完全重调度候选
评价与策略选择
小规模实验验证
```

阶段 F 之后的目标不是一次性加入所有复杂扰动，而是逐步增加订单取消场景的灵活性、策略适应性、实验可信度和工程可复现性。

## 1. 长期目标

长期目标：

```text
给定一个 FJSP-AGV 正常调度计划和不同类型的订单取消事件，
系统能够稳定提取动态状态，
按场景选择局部修复、完全重调度或混合修复策略，
在机器、AGV、工序顺序和冻结任务约束下生成可行方案，
用多指标评价效率、能耗和扰动，
并在多场景、多随机种子实验中形成可复现结论。
```

达到“灵活调度”的最低验收标准：

1. 支持不同取消时刻：早期、中期、后期。
2. 支持不同取消对象：短工件、长工件、关键路径工件、非关键工件。
3. 支持至少两类处理策略：局部修复、完全重调度。
4. 支持至少一类混合策略：局部修复失败或收益不足时触发完全重调度。
5. 每次调度后都能通过机器、AGV、工序顺序、冻结任务和取消任务排除检查。
6. 每次实验都输出原始指标和综合指标 `Y`。
7. 研究结论必须来自多场景和多随机种子汇总，不能来自单次 smoke。

## 2. 总体阶段划分

阶段 F 之后建议按 G-N 推进：

| 阶段 | 名称 | 核心问题 |
|---|---|---|
| G | 场景库与实验基准扩展 | 取消事件是否覆盖不同时间、不同订单和不同难度 |
| H | 混合修复策略 | 什么时候只做局部修复，什么时候触发完全重调度 |
| I | 多订单连续取消 | 连续取消时状态能否正确回放和重复修复 |
| J | 订单变更统一框架 | 订单取消和新订单插入能否统一为任务集合变化 |
| K | 自适应策略选择 | 权重、阈值或策略选择能否随场景自动调整 |
| L | 大规模与统计验证 | 结论是否在更多实例和随机种子下稳定 |
| M | 多扰动扩展接口 | 以后如何接机器故障和 AGV 故障，但不混乱当前主线 |
| N | 论文级整理与复现包 | 是否能形成论文实验、图表、消融和复现说明 |

## 3. 阶段 G：场景库与实验基准扩展

目标：

```text
建立可复现的订单取消场景库，让实验不只依赖单个 Mk01 smoke。
```

范围：

1. 多个取消时刻：早期、中期、后期。
2. 多类取消订单：短工件、长工件、关键路径工件、非关键工件。
3. 多个随机种子。
4. 多个小中规模样例数据。

建议新增：

```text
configs/order_cancellation_scenario_library.yaml
src/cancellation/build_order_cancellation_scenarios.m
tests/test_order_cancellation_scenario_library.m
docs/00_system_overview/stage_g_scenario_library_plan.md
```

验收标准：

1. 能从配置生成一组取消场景。
2. 每个场景都有 `scenario_id`、`cancel.job_id`、`cancel.cancel_time`、`cancel.policy`。
3. 场景不写死本机绝对路径。
4. 能统计每类场景数量。
5. 不运行正式实验，只验证场景生成。

停止条件：

```text
如果无法稳定判断关键路径工件或工件长度，先只做早期/中期/后期和随机 job_id，不强行定义关键路径。
```

## 4. 阶段 H：混合修复策略

目标：

```text
在局部修复和完全重调度之间建立触发规则，形成第一版混合策略。
```

思路：

1. 先生成局部修复候选。
2. 如果局部修复不可行，触发完全重调度。
3. 如果局部修复可行但 `Cmax_delta`、能耗或空闲浪费超过阈值，触发完全重调度。
4. 如果两个候选都可行，继续使用阶段 E 的 `Y` 选择。

建议新增：

```text
src/cancellation/select_hybrid_cancellation_policy.m
tests/test_order_cancellation_hybrid_policy.m
docs/00_system_overview/stage_h_hybrid_policy_plan.md
```

验收标准：

1. 能处理局部修复可行、完全重调度可行。
2. 能处理局部修复不可行、完全重调度可行。
3. 能处理两个候选都不可行。
4. 触发规则来自 `config`，不写死在函数里。
5. 输出选择原因，例如 `local_stable_enough`、`complete_better_Y`、`local_infeasible_trigger_complete`。

停止条件：

```text
如果阈值缺少实验依据，先使用可解释的默认阈值，并在文档中标记为待阶段 L 验证。
```

## 5. 阶段 I：多订单连续取消

目标：

```text
支持多个订单按时间顺序连续取消，验证状态回放和重复重调度能力。
```

范围：

1. 多个取消事件按 `cancel_time` 排序。
2. 每次取消后以上一次候选计划作为新基线。
3. 已完成任务继续冻结。
4. 已取消订单不能回流。

建议新增：

```text
src/cancellation/run_sequential_order_cancellations.m
tests/test_order_cancellation_sequential_events.m
docs/00_system_overview/stage_i_sequential_cancellation_plan.md
```

验收标准：

1. 能处理至少 2 个连续取消事件。
2. 每次事件后都输出局部修复、完全重调度和最终选择。
3. 每次事件后都通过约束检查。
4. 后一次状态提取基于前一次选择后的计划。
5. 被早先取消的订单不会在后续计划中回流。

停止条件：

```text
如果连续事件中出现正在加工任务取消，先按 unsupported 记录，不扩展抢占或中断加工逻辑。
```

## 6. 阶段 J：订单变更统一框架

目标：

```text
把订单取消和新订单插入统一建模为任务集合变化，为更灵活的订单变更调度做准备。
```

范围：

1. 订单取消：从任务集合删除未完成任务。
2. 新订单插入：向任务集合增加新工件。
3. 两者共享状态提取、冻结任务、候选生成和评价框架。

建议新增：

```text
src/events/create_schedule_change_event.m
src/events/validate_schedule_change_event.m
docs/00_system_overview/stage_j_order_change_framework_plan.md
```

验收标准：

1. 能用统一事件结构表示 `cancel_order`。
2. 能预留 `insert_order`，但不强行实现完整插单算法。
3. 订单取消现有测试不回退。
4. 文档说明取消和插单的共同点与差异。

停止条件：

```text
如果插单需要改动 problem 数据结构过大，先只完成事件接口设计，不进入插单解码实现。
```

## 7. 阶段 K：自适应策略选择

目标：

```text
让策略选择不只依赖固定权重，而能根据场景特征调整权重或选择规则。
```

可选路线：

1. 规则式自适应：根据取消时刻、剩余任务数、空闲窗口大小调整权重。
2. 数据驱动分类器：用实验结果学习什么时候选择局部修复或完全重调度。
3. 强化学习：作为后续创新方向，用于策略选择或搜索预算分配，不直接替代底层解码器。

第一版建议先做规则式自适应，不直接上强化学习。

建议新增：

```text
src/cancellation/extract_cancellation_features.m
src/cancellation/adapt_evaluation_weights.m
tests/test_order_cancellation_adaptive_weights.m
docs/00_system_overview/stage_k_adaptive_strategy_plan.md
```

验收标准：

1. 能提取场景特征，例如剩余工序数、取消时刻比例、被取消任务数量、冻结任务比例。
2. 能根据特征生成 `config.weights`。
3. 权重变化可解释。
4. 固定权重策略仍可作为 baseline。
5. 不把强化学习作为阶段 K 的必要完成条件。

停止条件：

```text
如果实验样本太少，不训练模型；只做规则式权重和文档说明。
```

## 8. 阶段 L：大规模与统计验证

目标：

```text
验证订单取消策略在更多实例、更多场景和更多随机种子下是否稳定。
```

范围：

1. 多个 FJSP-AGV 数据实例。
2. 多类取消场景。
3. 每个场景多个随机种子。
4. 汇总均值、标准差、胜率和不可行率。

建议新增：

```text
scripts/run_order_cancellation_benchmark.m
src/cancellation/summarize_order_cancellation_benchmark.m
docs/00_system_overview/stage_l_benchmark_plan.md
```

验收标准：

1. 每个实例都有配置记录。
2. 每个场景都有原始 seed 结果。
3. 汇总表包含 `Cmax_delta`、`SD`、`TD`、`energy_delta`、`Y`、策略胜率。
4. 图表或表格可复现。
5. 结论说明统计范围，不声称全局最优。

停止条件：

```text
如果运行时间超过可接受范围，先缩小实例规模或 seed 数，记录预算，不强行跑大实验。
```

## 9. 阶段 M：多扰动扩展接口

目标：

```text
为机器故障、AGV 故障等资源扰动预留接口，但保持订单取消主线清晰。
```

范围：

1. 只定义事件接口和文档边界。
2. 不把机器故障逻辑混入订单取消稳定实现。
3. 不修改 `raw_code/`。

建议新增：

```text
docs/00_system_overview/stage_m_multi_disruption_interface_plan.md
```

验收标准：

1. 文档说明订单取消是任务集合变化。
2. 文档说明机器故障是资源可用性变化。
3. 文档说明 AGV 故障是运输资源可用性变化。
4. 三类事件的状态提取和修复入口边界清晰。

停止条件：

```text
如果机器故障项目目录或代码边界不清楚，停止，不在当前订单取消主线中实现机器故障。
```

## 10. 阶段 N：论文级整理与复现包

目标：

```text
把订单取消动态重调度整理成可写论文、可复现实验和可答辩的项目材料。
```

范围：

1. 问题定义。
2. 方法流程图。
3. 算法伪代码。
4. 实验配置。
5. 指标定义。
6. 结果表格。
7. 消融分析。
8. 局限与未来工作。

建议新增：

```text
docs/paper/order_cancellation_method.md
docs/paper/order_cancellation_experiment_protocol.md
docs/paper/order_cancellation_results_summary.md
```

验收标准：

1. 能说明创新点：订单取消下的冻结-删除-候选生成-多指标选择框架。
2. 能说明不是全局最优证明。
3. 能说明和机器故障、新订单插入、强化学习的边界。
4. 每张表或图都能追溯到 `outputs/`。
5. 复现实验命令和配置齐全。

停止条件：

```text
如果阶段 L 的实验还不足以支撑结论，先补实验，不提前写强结论。
```

## 11. 目标分层

后续目标可以分为三层：

### 11.1 可运行

最低要求：

1. 每个场景能生成可行候选。
2. 每个候选能被评价。
3. 输出能落盘。

### 11.2 可比较

中等要求：

1. 每个场景有局部修复和完全重调度对比。
2. 每个指标有统一口径。
3. 多随机种子能汇总。

### 11.3 可研究

论文级要求：

1. 场景覆盖足够。
2. 结论来自统计汇总。
3. 有 baseline、消融和局限说明。
4. 可复现配置完整。

## 12. Scope 控制

后续推进时仍遵守：

1. 不修改 `raw_code/`。
2. 不一次性重构全项目。
3. 每次只进入一个阶段。
4. 每个阶段先写契约和测试，再实现算法。
5. MATLAB 实验和 `outputs/` 生成前需要确认。
6. 不把单次 smoke 当作研究结论。
7. 不把全局最优证明作为项目完成条件。

## 13. 当前建议的下一步

当前 A-E 已有第一版链路，F 已有工作计划。因此建议下一步顺序是：

1. 先完成阶段 F：早期/中期/后期取消小规模实验。
2. 再进入阶段 G：扩展场景库。
3. 再进入阶段 H：混合修复策略。
4. 等实验样本足够后，再考虑阶段 K 的自适应策略选择。

不建议现在直接做强化学习或机器故障，因为当前更需要先证明订单取消场景本身的实验闭环稳定。
