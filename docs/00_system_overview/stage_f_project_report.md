# 阶段 F 项目报告：小规模订单取消实验

本文档记录阶段 F 小规模实验的实际运行结果和第一版分析结论。结论仅基于本次 `Mk01` 小样例、多随机种子输出，不代表大规模实验结论，也不声称全局最优。

## 1. 实验目标

阶段 F 的目标是验证订单取消第一版闭环：

```text
取消场景配置
  -> 阶段 B 状态提取
  -> 阶段 C 局部修复候选
  -> 阶段 D 完全重调度候选
  -> 阶段 E 指标评价和策略选择
  -> 多随机种子小规模汇总
```

本阶段关注：

1. 早期取消、中期取消、后期取消是否都能运行。
2. 每个场景是否同时报告局部修复和完全重调度。
3. 每个场景是否报告调度约束检查结果。
4. 多随机种子结果能否汇总。
5. 是否能基于汇总结果形成第一版小规模结论。

### 4.2 中期取消指标

`middle_cancel` 是本次唯一可评价的场景。三个 seed 的结果一致：

| 指标 | 局部修复 | 完全重调度 |
|---|---:|---:|
| `Cmax_delta` | `0.000000` | `-3.000000` |
| `SD` | `0.000000` | `3.333333` |
| `TD` | `0.000000` | `2.000000` |
| `energy_delta` | `-11.600000` | `-13.633333` |
| `Y` | `0.500000` | `0.500000` |

解释：

1. 完全重调度在中期取消中平均使 `Cmax` 降低 `3.000000`，比局部修复更有利于缩短最大完工时间。
2. 完全重调度的 `energy_delta` 为 `-13.633333`，比局部修复的 `-11.600000` 更低，说明在当前简化能耗口径下能耗改善更大。
3. 完全重调度同时带来更大的扰动：`SD = 3.333333`，`TD = 2.000000`。
4. 综合指标 `Y` 两者相同，均为 `0.500000`。
5. 因 `Y` 相同，阶段 E 的第一版 tie-break 规则选择更保守的 `local_repair`。

## 5. 小规模结论

基于 `20260621_094024` 的阶段 F 小规模实验，本次共运行 9 个场景-seed 组合，覆盖 `early_cancel`、`middle_cancel`、`late_cancel` 三类取消场景。

第一版结论如下：

1. 本次只有 `middle_cancel` 场景形成可评价候选，且 3 个 seed 均可评价。
2. `early_cancel` 和 `late_cancel` 均未形成可行候选，输出选择原因均为 `no_feasible_candidate`。
3. 在 `middle_cancel` 中，完全重调度稳定降低 `Cmax`，并在当前能耗口径下降低更多能耗。
4. 在 `middle_cancel` 中，完全重调度带来更大 `SD` 和 `TD`，说明效率收益伴随计划扰动增加。
5. 在 `middle_cancel` 中，两类候选的 `Y` 相同，因此第一版策略选择按 tie-break 规则选择 `local_repair`。
6. 当前结果不能证明局部修复整体优于完全重调度，只能说明在本次权重、归一化和 tie-break 规则下，中期取消最终选择了局部修复。

## 6. 不可行案例说明

`early_cancel` 和 `late_cancel` 的 6 次运行均显示：

```text
selected_reason = no_feasible_candidate
```

同时，`seed_results.csv` 中对应候选的可行性和约束检查字段均为 `0`，指标为 `NaN`。

当前 CSV 没有保存 `rejectedReasons` 的具体文本，因此本报告只能确认这些案例没有形成可评价候选，不能进一步断言不可行来自机器冲突、AGV 冲突、工序顺序、冻结一致性或取消任务回流中的哪一类。

后续应扩展 `seed_results.csv`，把 `local_rejectedReasons` 和 `complete_rejectedReasons` 写入输出，便于定位不可行原因。

## 7. 局限

本次阶段 F 结果有以下局限：

1. 只使用 `data_sample/Mk01.fjs` 小样例。
2. 只使用 3 个取消比例：`0.25`、`0.50`、`0.75`。
3. 只使用 3 个随机种子：`1`、`2`、`3`。
4. 当前脚本使用样例基线计划，不是大规模正式实验。
5. 完全重调度第一版使用确定性 first-choice 染色体，不是正式 NSGA-II 长搜索。
6. 能耗结论依赖当前简化能耗口径。
7. 当前输出没有记录详细不可行原因文本。
8. 不包含机器故障。
9. 不包含新订单插入。
10. 不包含多个订单连续取消。
11. 不包含强化学习。
12. 不包含全局最优证明。

因此，本报告是阶段 F 小规模实验报告，不是最终论文级实验结论。

## 9. 后续建议

建议下一步优先做：

1. 把 `rejectedReasons` 写入 `seed_results.csv`，定位早期和后期取消不可行原因。
2. 增加更多样例和更大规模数据集。
3. 增加更多随机种子。
4. 对完全重调度接入正式 independent NSGA-II 搜索，而不是只用 first-choice 候选。
5. 重新讨论 `Y` 的权重和 tie-break 规则，避免效率收益和扰动惩罚完全抵消后只能保守选择。
6. 在阶段 G 之后再扩展连续取消、新订单插入、机器故障或强化学习。
