# 第 18 步：编码-解码应用理解总览

## 1. 为什么编码-解码是调度算法核心

智能调度算法表面上在跑 NSGA-II、遗传算法、强化学习或其他搜索方法，但它真正操作的通常不是一张已经排好的甘特图，而是一串“决策表达”。

这串决策表达就是编码。

解码的作用，是把这串编码解释成一个现实中能执行的调度方案。

所以调度优化的核心链条是：

```text
现实调度问题
-> 需要做哪些决策
-> 把决策编码成算法能搜索的形式
-> 用解码规则还原成可执行调度
-> 用评价函数判断方案好坏
-> 算法继续搜索更好的编码
```

如果只看算法名字，很容易觉得“这个项目是在用 NSGA-II”。  
但更本质的是：

```text
这个项目把 FJSP-AGV 调度问题编码成 OS / MS / AS / SS，
再通过 sorting.m 解码成机器和 AGV 的时间轴。
```

以后换一个新课题，也要先问：

```text
这个问题的调度决策是什么？
这些决策怎么编码？
解码时必须满足哪些现实约束？
目标函数依赖哪些解码结果？
```

## 2. 当前项目的编码表达了哪些决策

当前 FJSP-AGV 项目里，一条染色体不是一个单独数字，而是几类调度决策拼在一起。

| 编码 | 决策问题 | 通俗理解 |
|---|---|---|
| `OS` | 工序顺序 | 哪个工件的下一道工序先进入调度 |
| `MS` | 机器选择 | 这道工序放到哪台候选机器上加工 |
| `AS` | AGV 选择 | 这道工序由哪辆 AGV 搬运 |
| `SS` | 速度选择 | AGV 空载/负载运输时用哪个速度档 |

也就是说，这条染色体同时回答了四类问题：

```text
先做谁？
在哪台机器做？
谁来搬？
用多快的速度搬？
```

算法搜索时，不是直接移动机器或 AGV，而是在改变这些编码。  
编码变了，解码后的调度过程也会变，最后 makespan 和 energy 也会变。

## 3. sorting.m 解码时解决了哪些现实约束

`sorting.m` 的作用不是简单把数字翻译成文字，而是把编码变成现实调度过程。

它在解码时要处理很多现实约束：

| 现实约束 | sorting.m 要处理什么 |
|---|---|
| 工序先后约束 | 同一个工件的后一道工序不能早于前一道工序 |
| 机器占用约束 | 同一台机器同一时间只能加工一个工序 |
| 候选机器约束 | 一道工序只能从允许的候选机器里选 |
| AGV 占用约束 | 同一辆 AGV 同一时间只能执行一个运输/充电事件 |
| 空载运输 | AGV 要先从当前位置移动到工件所在位置 |
| 负载运输 | AGV 再把工件送到目标机器 |
| 末道卸载 | 工件最后一道工序完成后还要送到卸载站 |
| 速度影响 | 速度会影响运输时间和 AGV 能耗 |
| 电量约束 | 电量不足时要安排充电 |
| 插空调度 | 工序可以插入机器已有时间轴的空闲段 |

所以 `sorting.m` 更像一个离散事件调度仿真器：

```text
读一段编码
安排一次运输
安排一次加工
更新时间轴
更新 AGV 电量
继续读下一段编码
```

最终它得到的不是一个抽象分数，而是一套机器和 AGV 的时间安排。

## 4. 这套思路以后怎么迁移到新课题

以后看到新智能调度课题时，不要先问“用了什么算法”，先按下面顺序问。

### 4.1 调度对象是什么

先找系统里有哪些实体：

```text
工件？
工序？
机器？
AGV？
工人？
机器人？
仓库？
充电站？
能源设备？
```

当前项目是：

```text
工件 + 工序 + 机器 + AGV + 速度 + 电量
```

### 4.2 决策变量是什么

再问系统要做哪些选择：

```text
谁先做？
在哪做？
谁运输？
走哪条路？
用什么速度？
什么时候充电？
```

当前项目把这些选择压缩成：

```text
OS / MS / AS / SS
```

### 4.3 编码怎么表达这些决策

编码不是越复杂越好，而是要能让算法搜索。

一个好编码至少要回答：

```text
长度怎么确定？
每个位置代表什么？
取值范围是否合法？
算法交叉变异后还容易保持可行吗？
```

### 4.4 解码怎么保证现实可行

很多约束不一定直接写在编码里，而是在解码时处理。

例如当前项目：

```text
OS 只说工件出现顺序
sorting.m 再判断这是该工件第几道工序
MS 只说选第几个候选机器
sorting.m 再映射成真实机器编号
SS 只说速度档位
sorting.m 再计算运输时间和能耗
```

这说明：

```text
编码表达决策。
解码负责把决策变成可行调度。
```

### 4.5 评价函数依赖哪些解码结果

目标值不是凭空算的。

当前项目里，`fitness.m` 依赖 `sorting.m` 得到：

```text
机器时间轴
AGV 时间轴
工件卸载完成时间
AGV 电量变化
```

然后才计算：

```text
makespan
machine energy
AGV energy
total energy
```

以后看新课题，也要问：

```text
目标函数需要解码器输出什么？
如果解码器没记录这个东西，目标函数怎么算？
```

## 5. 哪些内容要看代码，哪些只需要理解应用逻辑

你未来不是一定要逐行重写算法，所以不需要一开始就陷入所有代码细节。

更合适的层次是：

| 内容 | 是否必须逐行看代码 | 你需要掌握什么 |
|---|---|---|
| 调度对象 | 不需要 | 系统里有哪些实体 |
| 决策变量 | 不需要逐行 | 算法到底在决定什么 |
| 编码结构 | 需要看一点代码和文档 | 染色体分几段，每段含义是什么 |
| 解码主流程 | 需要看核心逻辑 | 编码如何变成机器/AGV 时间轴 |
| 复杂插空细节 | 不必一开始逐行 | 先知道它解决机器空闲时间插入问题 |
| 电量和充电细节 | 按需要看 | 知道这是 AGV 可行性和能耗的一部分 |
| 目标函数公式 | 需要理解 | makespan 和 energy 从哪些时间轴/记录来 |
| 搜索算法细节 | 后续再看 | 先知道算法在改染色体，不是直接改工厂 |

当前阶段最值得你掌握的是：

```text
调度对象 -> 决策变量 -> 编码 -> 解码 -> 评价
```

不是一上来背 `sorting.m` 每一行。

## 6. 当前项目给你的可迁移模板

以后看到一个新调度论文，可以套这个模板问：

| 问题 | 当前 FJSP-AGV 示例 |
|---|---|
| 调度对象是什么？ | 工件、工序、机器、AGV、速度、电量 |
| 决策有哪些？ | 工序顺序、机器选择、AGV 选择、速度选择 |
| 编码是什么？ | `OS / MS / AS / SS` |
| 解码器是谁？ | `sorting.m` |
| 解码输出什么？ | 机器时间轴、AGV 时间轴、电量记录、卸载完成时间 |
| 评价什么？ | makespan 和 total energy |
| 算法搜索什么？ | 搜索更好的染色体 |

这个模板比“记住某个算法名字”更重要。

因为不同论文可能换成：

```text
机器 + 工人
机器 + 机器人
AGV + 路径
多工厂
动态订单
低碳/能耗/维护约束
```

但核心问题仍然是：

```text
决策怎么编码？
编码怎么解码成可行方案？
方案怎么评价？
算法怎么搜索？
```

## 7. 本步结论

编码-解码不是为了炫技，也不是为了让你死磕代码。

它的意义是帮助你看懂：

```text
现实调度问题如何被算法处理。
```

当前项目里：

```text
OS / MS / AS / SS 负责表达决策。
sorting.m 负责把决策落成调度。
fitness.m 负责评价调度好坏。
NSGA-II 负责不断搜索更好的决策组合。
```

这条理解链路，才是以后迁移到新智能调度课题时最有价值的部分。
